# Contributing Guide

이 프로젝트는 가상검증팀의 해석 의뢰 현황판을 공동으로 개발하기 위한 저장소입니다. 원활한 협업을 위해 다음 지침을 따라 주세요.

## 브랜치 전략

- `main` 브랜치는 항상 배포 가능한 상태를 유지합니다.
- 새로운 기능이나 개선 작업은 `feature/기능명` 형태의 별도 브랜치에서 진행하세요.
- 버그 수정은 `fix/버그명` 브랜치를 사용합니다.
- 완료된 기능은 Pull Request를 통해 `main` 브랜치에 병합하며, 코드 리뷰를 거쳐 승인 후 머지합니다.

## 커밋 메시지

- 커밋 메시지는 한글 또는 영어로 작성하되, 변경 내용이 명확히 드러나도록 합니다.
- 형태는 다음과 같이 권장합니다.
  - `feat: 의뢰 수정 이력 관리 기능 추가`
  - `fix: 진행률 계산 오류 수정`
- 커밋 본문에는 변경 배경과 영향을 간단히 기술합니다.

## 코드 작성 지침

- Power FX 코드의 경우 [MS]개발 프로젝트의 지침을 따릅니다. 즉, **정확한 함수 사용**, **비위임 함수 최소화**, **가독성 확보**를 중요하게 생각합니다.
- 새로운 화면이나 컴포넌트를 추가할 때는 `/docs` 폴더에 설계 문서를 먼저 작성하고 논의하는 것을 권장합니다.
- SharePoint 리스트 컬럼 추가 시 해당 컬럼의 역할과 데이터 형식을 README 또는 문서에 업데이트하세요.
- UI 레이블과 메시지는 사용자 이해를 돕기 위해 한글로 작성하되, 내부 변수명과 함수명은 영어로 통일합니다.

## 리뷰 프로세스

- PR 제출 시 **변경된 기능**, **테스트 방법**, **예상되는 영향**을 상세히 적어 주세요.
- 리뷰어는 코드의 정확성뿐 아니라 MS 개발 지침 준수 여부(위임 가능성, 성능 등)도 확인합니다.
- 수정 요청을 받으면 가급적 신속하게 반영하고, 추가 설명이 필요한 경우 커멘트를 통해 소통하세요.

## 문서화

- 프로젝트의 요구사항이나 설계 변경 사항은 `/docs` 폴더에 Markdown 파일로 저장합니다.
- README는 전체 프로젝트 소개와 기본 사용 방법을 안내하는 문서이므로, 기능 추가/변경 시 같이 업데이트해 주세요.

## 기타

- 개인정보나 회사 기밀 정보는 저장소에 업로드하지 마세요.
- 외부 라이브러리 사용 시 라이선스를 확인하고, 필요한 경우 `/licenses` 폴더에 관련 문서를 포함합니다.
